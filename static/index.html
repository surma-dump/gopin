GoPin Tool-less version pinning for Go

Ever used a package by some idiot who doesn't maintain backwards-compatibility? 

import (
    "github.com/surma/httptools"
)

↓

Pin it to the commit you have developed against!

import (
    "gopin.org/github.com/surma/httptools/ef4a90a9c9a90152e154fa8ba46cc8858f3476ec"
)

Important facts

The good:

  • GoPin is open source and MIT licensed
  • No local tools are needed
  • Your projects remain go get-able
  • No additional strain on the maintainer’s or the user’s side

The bad:

  • Only Github is supported (others soon™)
  • Only full commit hashes are supported (short hashes and tags soon™)

The ugly:

  • GoPin will create a new clone for each pinned import in your $GOPATH
  • GoPin will not work with private repositories (but neither does go get)

How does it work?

GoPin acts as a man in the middle between the go tool and Github. It parses the data stream while the repository is being cloned. When the position of the master branch is being transmitted, GoPin injects the given hash.

Beacuse gopin.org is not a natively supported VCS hoster, the custom remote import feature is used to make the go tool behave just the same.
